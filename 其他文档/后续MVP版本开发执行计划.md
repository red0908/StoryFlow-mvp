## 2. 当前状态与目标

### 2.1 当前已完成（首轮 MVP）

- 技术栈：Vite + React + TypeScript + Zustand + Less/Tailwind，纯前端、剧本 JSON 驱动。
- 流程：首页 → 创建角色 → 选择相亲对象（5 选 1）→ 游戏剧情页 → 结局页（按好感度三种结局）。
- 剧本：`story.json` 当前仅 1 个节点（c1_node1），选项点击后无下一节点即跳转结局，形成单节点闭环。
- 结局：affection ≥80 → 灵魂伴侣；50~79 → 略有遗憾；<50 → 分道扬镳。

### 2.2 后续 MVP 目标

| 目标项       | 说明 |
|--------------|------|
| 剧本完整可玩 | 4 章剧情，每章 3~4 个节点，总节点数 12~15；从见面到结局有完整叙事线。 |
| MBTI 差异化  | 选项效果使用 valueByMbti 体现人格差异；关键节点可选条件（如好感度）控制选项出现。 |
| 视觉与体验   | 替换占位图为 AIGC/素材；按章节或结局切换背景；移动端基本可用。 |
| 稳定上线     | 部署至 Vercel/静态托管，全分支测试通过，可分享链接或二维码。 |

## 3. 阶段划分与优先级

### 阶段一：剧本内容完整（最高优先级）

**目标**：`story.json` 扩充为 4 章、每章 3~4 节点，节点间通过 `nextNode` 串联，最后一章结束无下一节点时进入结局。

| 任务 | 说明 | 参考文档 |
|------|------|----------|
| 第一章完整 | 3~4 个节点：见面 → 寒暄/工作/兴趣 → 结束离场或约定下次；每节点 2~3 选项，含 effects（value 或 valueByMbti）。 | 需求文档 6.1、6.2；技术架构 7.2 |
| 第二章完整 | 3~4 个节点：第二次约会（如公园/餐厅），话题深入，选项影响好感。 | 同上 |
| 第三章完整 | 3~4 个节点：关系推进或矛盾点，可加入 1~2 处 condition（如好感≥60 才出现某选项）。 | 游戏剧情模块设计 6.3、6.4 |
| 第四章完整 | 3~4 个节点：收束到「是否表白/确认关系」等，最后一节点无 nextNode，由 GamePage 跳转 /ending。 | 结局页设计、技术架构 6.4 |

**产出**：完整 `story.json`（约 12~15 个节点），可从头玩到尾并根据好感度看到三种结局之一。

### 阶段二：选项条件与 MBTI 效果完善（高优先级）

**目标**：剧本中关键选项具备条件与差异化效果，与类型定义一致。

| 任务 | 说明 | 技术实现 |
|------|------|----------|
| 条件过滤 | 部分选项带 `condition`（如 `affection >= 60`），仅满足时展示。 | GamePage 中 filterOptions + evaluateCondition（已支持） |
| MBTI 效果 | 重要选项使用 `valueByMbti` 替代单一 `value`，体现不同人格反应。 | AffectionEffect.valueByMbti + applyEffects（已支持） |
| 节点 ID 规范 | 统一命名（如 c1_node1、c2_node1），便于维护与排查跳转。 | story.json 与文档一致 |

**产出**：剧本中至少 3~5 处条件选项、5~8 处 MBTI 差异化好感效果；无跳转错误。

### 阶段三：视觉资源与界面优化（中优先级）

**目标**：替换占位图、按章节/结局区分氛围，界面美观可对外展示。

| 任务 | 说明 | 资源位置 |
|------|------|----------|
| 背景图 | 咖啡厅/公园/餐厅/海边等 4 章场景（或复用 1~2 张）+ 三种结局背景（可选）。 | public/background/ |
| 头像/立绘 | 按性别+MBTI 的 avatar 图（如 avatar_female_intj.png），与 gameConfig 一致。 | public/person_img/ |
| 章节/结局氛围 | 按 chapter 或结局类型切换背景或滤镜（暖色/冷色）。 | GamePage、EndingPage |
| 基础动效 | 按钮 hover、选项点击反馈（已有可保留），无复杂动画。 | 各页面 Less / Tailwind |

**产出**：关键页面有统一视觉风格，无大面积占位色块。

### 阶段四：移动端与部署（中低优先级）

**目标**：小屏可读可点、部署后可分享。

| 任务 | 说明 | 备注 |
|------|------|------|
| 响应式 | 剧情框、选项区、状态栏在 1280px 以下或竖屏下不溢出、可滚动。 | 技术架构 9.1 当前以 PC 为主，可逐步放宽 min-width |
| 触控 | 选项按钮足够大、间距合理，避免误触。 | 设计文档一致 |
| 部署 | Vercel/Netlify 等静态托管，SPA 回退到 index.html。 | 技术架构 8.2 |
| 测试 | 全分支走通（至少 2~3 条路径）、三种结局均能触发。 | 成功标准 |

**产出**：线上可访问链接，手机浏览器可完成一轮完整游戏。

## 4. 时间与分工建议（兼职友好）

- **总工时预估**：在首轮 26 工时之外，后续 MVP 约 **15~20 工时**（可按周拆分）。
- **节奏**：工作日 1~2h/天做剧本或小优化，周末 4~6h 做视觉或联调。

| 周 | 侧重 | 用户侧 | 开发/Cursor 侧 |
|----|------|--------|----------------|
| 第 1 周 | 剧本 | 按《完整剧本案例大纲》填写 4 章节点文本与选项 | 协助校验 nextNode 串联、effects/condition 格式，修 bug |
| 第 2 周 | 条件与效果 | 标注哪些选项要 condition、哪些用 valueByMbti | 确认过滤与计算逻辑，补充 1~2 处条件示例 |
| 第 3 周 | 视觉 | 准备/生成背景与头像资源 | 接入资源、按章节/结局切换背景与头像 |
| 第 4 周 | 上线 | 全路径测试、记录问题 | 响应式与触控微调、部署与回退配置、修 bug |

## 5. 成功标准（后续 MVP 验收）

- [ ] 可从头（选人后）玩到结局，经历 4 章、每章至少 3 个节点。
- [ ] 选项会影响好感度，且至少部分选项因对方 MBTI 产生不同数值变化。
- [ ] 至少一处选项因好感度条件（如 ≥60）才出现。
- [ ] 三种结局（灵魂伴侣 / 略有遗憾 / 分道扬镳）均可通过不同选择触发。
- [ ] 关键界面有统一视觉（非纯占位色块），移动端可完成一轮流程。
- [ ] 已部署至公网，可分享链接或二维码。

## 7. 风险与应对

| 风险 | 应对 |
|------|------|

| 移动端布局问题 | 先保证功能与 PC，再单独排期做响应式与触控优化。 |
