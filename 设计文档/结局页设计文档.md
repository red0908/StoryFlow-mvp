# 结局页设计文档：MBTI 心动相亲局

---

## 1. 设计目标与模块概述

### 1.1 目标
结局页是玩家完成当前轮剧情后的收尾页面，负责根据好感度（及后续可扩展的关键选择）展示对应结局类型（匹配成功 / 略有遗憾 / 分道扬镳等），呈现结局名称、评语与氛围，并提供「重新开始」等操作，形成从选人 → 剧情 → 结局的完整流程闭环。

### 1.2 在整体流程中的位置
```
首页 → 创建角色 → 选择相亲对象（5选1） → 【游戏剧情 /game】 → 【结局页 /ending】
```
- **入口**：由游戏剧情页在「无下一节点」或「章节结束」时跳转至 `/ending`（当前阶段：单节点剧本下，点击选项后无下一节点即跳转结局）。
- **出口**：「重新开始」回到首页或选择对象页；可选「分享」预留（暂不可用）。
- **与 game 的关系**：结局页是 game 的下级页面，依赖 game 产生的 `opponent`、`affection`、`player` 等状态。

### 1.3 兼容范围
- **当前阶段**：优先兼容 **Web 端 PC**（桌面浏览器），与首页、创建角色、游戏剧情、视觉设计文档一致。
- 后续可扩展移动端布局与触控。

## 2. 路由与参数

### 2.1 路由规则
| 项目     | 说明 |
|----------|------|
| 路径     | `/ending` |
| 查询参数 | 可选 `type`（如 `type=success`），用于直链或调试；正常从 game 跳转时由页面根据 store 计算结局类型，可不依赖 URL。 |
| 示例     | `/ending` 或 `/ending?type=success` |

### 2.2 参数约定
- **type**（可选）：若存在，可优先用于展示指定结局类型（开发/测试用）；否则根据 `useGameStore` 中的 `affection` 计算结局类型。
- 进入页面时若 `opponent` 为空，视为异常（如直接访问 `/ending`），可重定向到选择对象页或首页。

## 3. 需求来源与结局类型定义

### 3.1 依据需求文档（2.5 结局系统、简化版执行计划）

- 第四章最后一个节点结束后，根据**好感度范围**及关键选择展示对应结局。
- 预设 3 种结局（先实现闭环，隐藏结局后续迭代）：

| 结局类型     | 好感度条件 | 说明 |
|--------------|------------|------|
| 匹配成功     | affection ≥ 80 | 双方性格契合，相亲成功。 |
| 略有遗憾     | 50 ≤ affection < 80 | 有好感但未进一步发展。 |
| 分道扬镳     | affection < 50 | 性格不合，相亲失败。 |

### 3.2 先走一个流程闭环（当前阶段）

- **剧本现状**：`story.json` 仅有一个节点（如 `c1_node1`），选项的 `nextNode` 指向的节点不存在。
- **约定**：game 页在「点击选项后无下一节点」时，**不再展示「本章未完待续」**，改为 **直接跳转 `/ending`**；ending 页根据当前 `affection` 判定并展示上述三种结局之一，提供「重新开始」按钮回到首页或选人页，形成 **选人 → 单节点剧情 → 结局** 的闭环。
- 后续剧本扩充后，可改为：仅当「第四章最后一节点结束」或「显式剧情节点指向结局」时再跳转 `/ending`；当前单节点方案保持不变即可兼容。

### 3.3 结局数据结构（可配置，便于后续扩展）

可在 `public/data/` 下增加 `endings.json`（或先写死在代码中），例如：

```json
{
  "success": {
    "id": "success",
    "title": "灵魂伴侣",
    "subtitle": "匹配成功",
    "comment": "你们性格相契，彼此理解，这次相亲圆满成功。",
    "bg": "/background/bg_ending_success.png"
  },
  "regret": {
    "id": "regret",
    "title": "略有遗憾",
    "subtitle": "有好感但未更进一步",
    "comment": "你们相处愉快，但似乎还差一点缘分。",
    "bg": "/background/bg_ending_regret.png"
  },
  "fail": {
    "id": "fail",
    "title": "分道扬镳",
    "subtitle": "相亲失败",
    "comment": "性格与观念差异较大，未能走到一起。",
    "bg": "/background/bg_ending_fail.png"
  }
}
```

- 评语可支持插值：`{opponent.name}`、`{player.description}` 等，与 game 页一致。
- 若暂无独立结局背景，可复用 `bg-01.png` 或同一背景加不同 CSS 滤镜（暖色/冷色）区分氛围（见视觉设计 3.2）。

## 4. 资源路径与文件清单

### 4.1 根路径说明
路径均相对于项目 `public` 目录，前端引用使用 `/background/xxx`、`/person_img/xxx`、`/data/xxx` 等形式，与 game 页、首页一致。

### 4.2 背景（`/public/background/`）

| 文件名               | 用途说明 |
|----------------------|----------|
| `bg-01.png`          | 可与 game 共用，作为结局页默认/通用背景 |
| `bg_ending_success.png` | 匹配成功氛围（暖色、浪漫）可选 |
| `bg_ending_regret.png`  | 略有遗憾氛围（中性、柔和）可选 |
| `bg_ending_fail.png`    | 分道扬镳氛围（冷色、灰蓝）可选 |
| `card.png`           | 评语卡片、按钮区卡片背景（与首页、game 统一） |

- **当前阶段**：可仅用 `bg-01.png` + 不同滤镜（如 success 暖色、fail 冷色）区分；待资源就绪后再切换为独立背景图。

### 4.3 人物与立绘（`/public/person_img/`）

| 用途           | 说明 |
|----------------|------|
| 对方头像/立绘  | 根据 `opponent.mbti` 与玩家性别选用对应 `avatar_*_*.png`，与 game 页规则一致 |
| 玩家 MBTI 展示 | 可选：在结局评语区展示双方 MBTI 与简短匹配分析（见需求文档、执行计划） |

### 4.4 数据（`/public/data/`）

| 文件名         | 用途 |
|----------------|------|
| `endings.json` | 可选；结局类型与标题、评语、背景路径配置 |
| `gameConfig.json` | MBTI 代表色等，供结局页标签、氛围一致 |

## 5. 页面结构与布局

### 5.1 整体布局（PC）

- **视口**：与 game 页一致，以 1920×1080 为参考，最小宽度约 1280px。
- **背景**：结局对应背景图铺满、居中覆盖（或同一背景 + 滤镜）；与视觉设计 3.2、4.2.5 一致。
- **内容区**：自上而下建议为**顶部标题区**、**中部评语/结果卡片区**、**底部操作按钮区**。

### 5.2 顶部标题区

| 内容       | 说明 |
|------------|------|
| 结局名称   | 大标题，如「灵魂伴侣」「略有遗憾」「分道扬镳」（或配置中的 title） |
| 副标题     | 可选，如「匹配成功」「相亲失败」 |
| 风格       | 大号字体、居中，与视觉设计 4.2.5 一致；可按结局类型使用不同主色（成功暖色、失败冷色）。 |

### 5.3 中部评语/结果卡片区

- **评语**：毛玻璃卡片（可复用 `card.png` 或半透明圆角容器），展示当前结局的 `comment` 文案；支持 `{opponent.name}` 等插值。
- **可选**：双方 MBTI、好感度最终值、简短匹配分析（如「ENFJ 与 INTJ 在价值观上较为契合」），与需求文档、执行计划「完善结局界面」一致；首版可先只做评语。
- **对方头像/立绘**：可选在卡片旁展示对方头像，与 game 页 `getOpponentAvatar` 规则一致。

### 5.4 底部操作按钮区

| 按钮         | 行为说明 |
|--------------|----------|
| 重新开始     | 清空或重置游戏状态（如 `resetGame`），跳转至首页 `/` 或选择对象页 `/blindDate`（由产品定）。 |
| 分享         | 预留入口，暂不可用；可置灰或隐藏。 |

- 按钮样式与首页、game 页统一：圆角、毛玻璃、hover 反馈（见视觉设计 4.3）。

## 6. 交互与逻辑

### 6.1 进入页面

1. 若 store 中 `opponent` 为空，重定向到 `/blindDate` 或 `/`，不展示结局内容。
2. 根据 `affection` 计算结局类型：
   - `affection >= 80` → 匹配成功（success）
   - `50 <= affection < 80` → 略有遗憾（regret）
   - `affection < 50` → 分道扬镳（fail）
3. 若 URL 带 `type` 且为调试用途，可优先使用 `type` 展示对应结局。
4. 加载结局配置（若使用 `endings.json`）或使用内置文案/背景，渲染标题、评语、背景。

### 6.2 文本插值

- 评语中的 `{opponent.name}`、`{opponent.mbti}`、`{player.description}` 等替换为 store 中的实际值，与 game 页 `interpolateText` 逻辑一致。
- 未定义占位符替换为空或保留原样。

### 6.3 重新开始

- 调用 `useGameStore.getState().resetGame()`（或等价逻辑），清空 opponent、currentNodeId、affection、flags 等。
- 跳转至首页 `navigate('/', { replace: true })` 或选择对象页 `navigate('/blindDate', { replace: true })`；若希望保留玩家角色仅重选对象，可跳转选人页并只重置 game 状态。

### 6.4 与 game 页的衔接（单节点闭环）

- 在 **GamePage** 中，当选项点击后 `nextNode` 在剧本中**不存在**时：
  - **原逻辑**：展示「本章未完待续」并停留当前节点。
  - **闭环逻辑**：改为 `navigate('/ending', { replace: true })`，不再展示占位提示。
- 这样当前仅有一个节点的剧本下，用户流程为：选人 → 进入 game → 阅读唯一节点 → 点击任一选项 → 执行好感度变化 → 跳转 ending → 根据好感度看到对应结局 → 点击「重新开始」回到首页/选人，形成闭环。

---

## 7. 与状态管理及本地存储的衔接

- **Zustand**：结局页只读 `useGameStore` 的 `opponent`、`affection` 及 `usePlayerStore` 的 `player`；「重新开始」时调用 `resetGame()`。
- **本地存储**：若 game 流程中有自动保存进度，重置游戏时需一并清除与 game 相关的 localStorage（或由 `resetGame` 内统一处理），避免下次进入 game 残留旧进度。

## 9. 与现有设计的一致性

- **色彩与风格**：MBTI 代表色、毛玻璃、圆角与《视觉设计》一致；结局标题与卡片按结局类型在暖色/冷色上微调。
- **字体与组件**：标题、按钮、卡片与首页、创建角色、game 模块统一。
- **资源复用**：背景、卡片、人物图与 game、首页共用同一套 public 资源。
