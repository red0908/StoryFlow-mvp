# 选择相亲对象模块设计文档：MBTI 心动相亲局

---

## 1. 设计目标与模块概述

### 1.1 目标
选择相亲对象模块是玩家在「创建角色」点击「确认创建」后的第一个功能页面，负责根据玩家性别随机生成 5 位异性候选人，以卡片形式展示姓名、年龄、职业、MBTI、一句话介绍（及可选立绘），供玩家 5 选 1 锁定本次相亲对象，并写入游戏状态，为进入四章剧情提供数据基础。设计上需与首页、创建角色页、视觉设计文档保持一致，保证流程顺畅、卡片展示清晰、资源引用统一。

### 1.2 在整体流程中的位置
```
首页（开始游戏） → 创建角色 → 选择相亲对象（5选1） → 四章剧情 → 结局
```
- **入口**：创建角色页底部「确认创建」提交成功后跳转至本模块（当前路由为 `/select`）。
- **前置依赖**：必须已完成角色创建，即 Redux / Zustand 及 localStorage 中已有完整 `Player`（含性别、年龄、职业、MBTI、一句话介绍）；本页根据 `Player.gender` 决定生成异性候选人（男→女性候选人，女→男性候选人）。
- **出口**：玩家点击某位候选人并确认后，将该候选人写入游戏状态（如 `gameSlice.opponent`），再跳转到剧情主界面（第一章）；若产品支持「随机匹配」则可直接锁定随机一位并进入剧情。

### 1.3 兼容范围
- **当前阶段**：优先兼容 **Web 端 PC**（桌面浏览器）。
- 后续可扩展移动端（卡片纵向滑动、触控友好的点击与滑动）。

## 2. 需求来源与字段定义

### 2.1 依据需求文档（2.3 选择相亲对象）

| 需求要点 | 说明 |
|----------|------|
| 候选人数量 | 系统随机生成 **5 位异性**候选人 |
| 性别匹配 | 玩家性别为男则生成女性候选人，为女则生成男性候选人 |
| 候选人字段 | 姓名、年龄（25～35 随机）、职业（5 种中随机）、MBTI（4 种中随机）、一句话介绍（根据 MBTI 及职业组合生成） |
| 选择行为 | 玩家点击其中一位即锁定该角色为本次相亲对象，进入剧情 |

### 2.2 候选人数据结构（与需求文档 5.2、types.ts 一致）

```typescript
interface Candidate {
  id: string;        // 唯一标识，如 c1～c5 或随机 id
  name: string;      // 来自姓名库（按玩家性别取异性 names）
  age: number;       // 25～35 随机整数
  job: string;       // 职业 ID，与 gameConfig.jobs[].id 一致
  mbti: 'ENFJ' | 'INTJ' | 'INFJ' | 'ENTJ';
  tagline: string;   // 一句话介绍，可生成如「{name}，{age}岁，{jobLabel}，{mbtiTagline}」
}
```

### 2.3 配置与数据来源
- **姓名 / 职业 / MBTI 池**：来自 `public/data/gameConfig.json`。
  - `names.male` / `names.female`：按玩家性别取**异性**姓名池，从中无重复抽取 5 个名字。
  - `jobs`：5 种职业，随机分配给 5 位候选人（可重复或尽量不重复，按产品定）。
  - `mbti`：4 种人格，随机分配（可重复），用于展示 MBTI 标签及生成 tagline。
- **一句话介绍生成规则**：可参考《姓名与选项库说明》，例如  
  `"{name}，{age}岁，{jobLabel}，{mbtiTagline}"`，其中 jobLabel、mbtiTagline 从 gameConfig 中取对应项的 label / tagline。
- **随机逻辑**：每次进入选择界面（或点击「换一批」若有）重新随机生成 5 人，确保新鲜感（需求 3.2）。

## 3. 资源路径与文件清单

### 3.1 根路径说明
路径均相对于项目 `public` 目录，前端引用使用 `/background/xxx`、`/person_img/xxx` 等形式，与创建角色、首页一致。

### 3.2 背景资源（`/public/background/`）

| 文件名      | 用途说明 |
|-------------|----------|
| `bg-01.png` | 选择对象页全屏背景（与首页、创建角色页共用，铺满视口 `background-size: cover`） |
| `card.png`  | 候选人卡片背景（与首页、创建角色卡片风格统一） |
| `card1.png` | 备用卡片样式，可用于悬停或选中态 |

### 3.3 人物与立绘资源（`/public/person_img/`）

- **MBTI 展示用头像**（与创建角色、首页共用）：  
  `avatar_male_enfj.png`、`avatar_female_intj.png`、`avatar_female_infj.png`、`avatar_male_entj.png` 等。  
  选择对象页若在卡片上展示 MBTI 氛围，可按候选人 MBTI 与性别选用对应头像作为卡片插图。
- **候选人立绘**（视觉设计 2.2.2、7）：  
  半身立绘 800×1000px 透明 PNG，命名如 `candidate_01_base.png`～`candidate_05_base.png`（及表情差分 `_smile`、`_surprise`、`_think`）。  
  选择页卡片可使用 **base** 作为缩略图（建议缩略尺寸约 200×250px），立绘与候选人编号的对应关系可在生成候选人时绑定（如第 1 位对应 candidate_01_base.png），或按 MBTI/职业做映射，MVP 阶段也可先用 MBTI 头像占位。

### 3.4 配置与数据（`/public/data/`）

| 文件名             | 用途 |
|--------------------|------|
| `gameConfig.json`  | `names`（male/female）、`jobs`、`mbti`，供随机生成候选人使用 |
| `candidates.json`  | 可选：预置候选人示例，若采用「纯随机生成」则仅作参考或测试用 |

## 4. 页面结构与布局

### 4.1 整体布局（PC）

- **视口**：以 1920×1080 为参考，可设最小宽度（如 1280px）与首页、创建角色页一致。
- **背景**：`bg-01.png` 铺满、居中覆盖。
- **内容区**：居中容器，可置于毛玻璃或与首页统一的卡片风格内。

### 4.2 自上而下区域划分

| 区域         | 内容说明 |
|--------------|----------|
| 顶部标题     | 主标题如「选择你的心动对象」、副标题如「在 5 位候选人中选择一位开始约会」 |
| 可选：我的信息 | 简短展示当前玩家角色（如 MBTI + 一句话介绍），与创建角色页一致，增强代入感 |
| 候选人卡片区 | 5 张候选人卡片，横向滑动或网格排列（见下） |
| 底部操作     | 「随机匹配」按钮（一键随机选一位并进入剧情）；可选「换一批」重新生成 5 人 |

### 4.3 候选人卡片设计要点（对齐视觉设计 4.2.3）

- **布局**：每张卡片建议包含：
  - **左侧**：小立绘或 MBTI 头像（约 200×250px 缩略图），无立绘时可用 `person_img` 中对应 MBTI 头像或占位图。
  - **右侧**：姓名、年龄、职业（展示 job label）、MBTI 标签（含代表色）、一句话介绍（tagline）。
- **卡片样式**：使用 `card.png` 或 `card1.png` 作底，圆角、毛玻璃与视觉文档统一；悬停轻微放大，点击后进入「选中态」或弹层确认。
- **MBTI 标签**：与创建角色、视觉文档一致，ENFJ 暖橙、INTJ 冷紫、INFJ 森林绿、ENTJ 酒红，便于快速识别性格。
- **交互**：点击卡片 → 可高亮当前卡片并弹出简单确认（如「确定选择 Ta 开始约会吗？」）→ 确认后写入 opponent 并跳转剧情；若不做二次确认，则点击即视为选中并跳转。

### 4.4 可选：横向滑动与网格

- **横向滑动**：5 张卡片横向排列，可左右滑动浏览，适合宽屏。
- **网格**：如 5 张卡片 2+3 或 1+2+2 排列，更适合中等宽度；需保证每张卡片可点击区域充足、信息不拥挤。

## 5. 交互与逻辑

### 5.1 进入本页

- **触发**：创建角色页「确认创建」校验通过后，写入 Player 并 `navigate('/select')`。
- **前置校验**：若未存在 Player（如直接访问 `/select`），可重定向至 `/create/myRole` 或首页。
- **初始化**：读取 `Player.gender`，请求或已缓存的 `gameConfig.json`，调用随机生成逻辑生成 5 位 `Candidate`，写入本地状态（如 React state 或 store 的「当前候选列表」），不写入 opponent 直至用户选择。

### 5.2 随机生成 5 位候选人

- **性别**：`Player.gender === 'male'` → 从 `names.female` 中抽 5 个名字；`'female'` → 从 `names.male` 中抽 5 个（无重复抽取，若池子不足 5 个则允许重复或扩充池子）。
- **年龄**：每位 25～35 的随机整数。
- **职业**：从 `jobs` 中随机分配（可 5 人各不同，或允许重复）。
- **MBTI**：从 `mbti` 中随机分配（可重复）。
- **id**：生成唯一 id，如 `c1`～`c5` 或 `candidate_${Date.now()}_${index}`。
- **tagline**：按约定模板拼接，如 `"{name}，{age}岁，{jobLabel}，{mbtiTagline}"`。

### 5.3 选择一位候选人

- **点击卡片**：标记当前点击的候选人为「待确认」或直接进入确认逻辑。
- **确认选择**：
  - 将当前选中的 `Candidate` 写入游戏状态（如 `gameSlice.setOpponent(candidate)` 或等价 store）。
  - 可选：同时初始化 `currentChapter: 1`、`currentNodeId` 为第一章首节点、`affection: 50`、`flags: {}`。
  - 跳转至剧情主界面（路由按项目约定，如 `/game` 或 `/story`）。
- **随机匹配**：点击底部「随机匹配」时，从当前 5 人中随机选 1 人，执行与「确认选择」相同的写入与跳转逻辑。

### 5.4 可选功能

- **换一批**：重新执行随机生成，更新 5 位候选人列表并刷新展示，不改变 Player。
- **返回**：提供「返回上一步」至创建角色页（仅修改角色时使用），需注意是否允许覆盖已保存的 Player。

## 6. 与状态管理及本地存储的衔接

- **Player**：只读。来自 `usePlayerStore` 或 Redux `playerSlice`，用于性别判断与展示「我的信息」；本页不修改 Player。
- **候选人列表**：仅本页使用，可放在组件 state 或临时 store，无需持久化；进入剧情后仅保留「选中的那一位」。
- **Game 状态**：选择确认后写入 `opponent`（及初始章节、节点、好感度等），供剧情页读取；若使用 Zustand/Redux 持久化或 localStorage 保存进度，需在「选择确认」时一并初始化并可选写入本地存储。

## 8. 与现有设计的一致性

- **色彩与风格**：主色、毛玻璃、圆角、MBTI 代表色与《视觉设计》文档一致（ENFJ 暖橙、INTJ 冷紫、INFJ 森林绿、ENTJ 酒红）。
- **字体与组件**：标题、卡片、按钮与首页、创建角色、视觉文档统一；卡片悬停/点击反馈（放大、高亮、发光）与视觉文档 4.2.3、4.3 一致。
- **资源复用**：背景、卡片、人物图与首页、创建角色页共用同一套 public 资源；候选人立绘按视觉文档规格与命名放置于 `person_img`（或项目约定目录）。
