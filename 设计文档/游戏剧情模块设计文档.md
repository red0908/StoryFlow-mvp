# 游戏剧情模块设计文档：MBTI 心动相亲局

---

## 1. 设计目标与模块概述

### 1.1 目标
游戏剧情模块是玩家在「选择相亲对象」确认创建后进入的核心玩法页面，负责按当前章节加载剧本节点、展示剧情文本与选项，执行选项效果（好感度变化、节点跳转），并维持与整体视觉、资源路径的一致性。当前阶段剧本仅有一个节点，优先完成**单节点**的完整交互与占位逻辑，为后续多节点、多章节扩展预留接口。

### 1.2 在整体流程中的位置
```
首页 → 创建角色 → 确认创建 → 选择相亲对象（5选1） → 【游戏剧情模块】 → 结局
```
- **入口**：选择相亲对象页确认后，跳转至 `/game?chapter=1`（或当前章节），并传入路由参数 `chapter` 表示当前章节。
- **出口**：本章节无下一节点时，可展示「本章结束」或占位提示；第四章结束后进入结局模块（后续迭代）。

### 1.3 兼容范围
- **当前阶段**：优先兼容 **Web 端 PC**（桌面浏览器），与首页、创建角色、视觉设计文档一致。
- 后续可扩展移动端（触控友好的选项按钮、适配小屏的布局）。

## 2. 路由与参数

### 2.1 路由规则
| 项目     | 说明 |
|----------|------|
| 路径     | `/game` |
| 查询参数 | `chapter`（当前章节，如 `chapter=1`） |
| 示例     | `/game?chapter=1` |

### 2.2 参数约定
- **chapter**：必选，取值 1～4，表示当前所处章节。进入页面时从 URL 读取，并与状态中的 `currentChapter` 同步；切换章节时更新 URL 与状态。
- 若未传 `chapter`，建议默认 `chapter=1`。

## 3. 需求来源与数据定义

### 3.1 依据需求文档（2.4 剧情互动、3.3 剧本驱动引擎）

- **顶部状态栏**：当前章节（如「第一章·开头」）、双方 MBTI 图标、当前好感度（数值或进度条）。
- **中部剧情框**：当前节点的描述文本，支持动态插入 `{opponent.name}`、`{opponent.mbti}`、`{player.xxx}` 等占位。
- **底部选项区**：2～3 个可点击选项按钮，点击后执行效果并跳转下一节点；若下一节点不存在则走单节点占位逻辑。

### 3.2 剧本节点结构（与 `public/data/story.json` 一致）

当前剧本为**数组**形式，每个元素为一个节点：

```json
{
  "id": "c1_node1",
  "chapter": 1,
  "text": "你坐在咖啡厅，等待 {opponent.name} 的到来。她/他穿着一件简单的衬衫，微笑着向你打招呼。",
  "options": [
    {
      "text": "直接开门见山，聊聊工作。",
      "nextNode": "c1_node2",
      "effects": [
        { "type": "affection", "operator": "add", "value": 2 }
      ]
    },
    {
      "text": "先夸赞对方的穿着，拉近距离。",
      "nextNode": "c1_node3",
      "effects": [
        {
          "type": "affection",
          "operator": "add",
          "valueByMbti": { "INTJ": 1, "ENFJ": 3, "INFJ": 2, "ENTJ": 1 }
        }
      ],
      "condition": { "type": "affection", "operator": "<=", "value": 60 }
    }
  ]
}
```

### 3.3 关键数据类型（与需求文档 5.2、现有 types 一致）

- **Player**：玩家性别、年龄、职业、MBTI、一句话介绍（来自角色创建与 store）。
- **Candidate（opponent）**：当前选择的相亲对象，含 name、mbti、job、tagline 等。
- **GameState**：opponent、currentChapter、currentNodeId、affection、flags。
- **StoryNode**：id、chapter、text、options；**StoryOption**：text、nextNode、effects、condition（可选）。

## 4. 资源路径与文件清单

### 4.1 根路径说明
路径均相对于项目 `public` 目录，前端引用使用 `/background/xxx`、`/person_img/xxx`、`/data/xxx` 等形式。

### 4.2 背景与卡片（`/public/background/`）

| 文件名      | 用途说明 |
|-------------|----------|
| `bg-01.png` | 剧情页全屏背景，铺满视口（可与首页、创建角色页共用） |
| `card.png`  | 剧情框或选项卡片背景，与首页、创建角色视觉统一 |

### 4.3 人物与立绘（`/public/person_img/`）

| 用途           | 说明 |
|----------------|------|
| 状态栏对方头像 | 根据 `opponent.mbti` 与性别选用对应 `avatar_*_*.png`（如 `avatar_female_intj.png`） |
| 玩家 MBTI 展示 | 仅展示 MBTI 图标/代表色，不展示立绘（与视觉设计一致） |

人物文件名规则与首页、创建角色模块一致（如 `avatar_male_enfj.png`、`avatar_female_intj.png` 等）。

### 4.4 剧本与配置（`/public/data/`）

| 文件名            | 用途 |
|-------------------|------|
| `story.json`      | 剧本节点数组，按 `id` / `chapter` 查找当前节点 |
| `gameConfig.json` | MBTI 的 label、tagline 等，供状态栏、插值展示 |

## 5. 页面结构与布局

### 5.1 整体布局（PC）

- **视口**：以 1920×1080 为参考，可设最小宽度（如 1280px）。
- **背景**：`bg-01.png` 铺满、居中覆盖（`background-size: cover`）。
- **内容区**：自上而下分为**顶部状态栏**、**中部剧情框**、**底部选项区**，与需求文档 2.4 一致。

### 5.2 顶部状态栏

| 内容           | 说明 |
|----------------|------|
| 章节标题       | 如「第一章·开头」，可从 `chapter` 与配置映射 |
| 玩家 MBTI      | 图标 + 人格类型（ENFJ/INTJ/INFJ/ENTJ），代表色见视觉设计 |
| 对方头像/信息  | 小头像（来自 `person_img`）+ 对方 MBTI 或姓名（可选） |
| 好感度         | 数值（0～100）或进度条，实时从 store 读取 |

布局建议：左侧章节标题，中间或右侧为双方 MBTI 与好感度，风格与视觉设计文档中的毛玻璃、圆角一致。

### 5.3 中部剧情框

- **内容**：当前节点的 `text`，经**文本插值**后展示（将 `{opponent.name}`、`{opponent.mbti}` 等替换为 store 中的实际值）。
- **容器**：可使用 `card.png` 或毛玻璃容器，保证可读性。
- **排版**：字号适中、行高舒适，可限制最大宽度以利阅读。

### 5.4 底部选项区

- **内容**：当前节点的 `options` 中，**通过条件过滤后**的选项（见下文 6.3）。
- **形式**：2～3 个按钮或卡片，每项展示选项文案；点击后执行效果并尝试跳转下一节点。
- **单节点占位**：若 `nextNode` 在剧本中不存在，不跳转，可提示「本章未完待续」或「当前仅开放单节点演示」并保持在本节点（或统一跳转至占位节点由产品定）。

## 6. 交互与逻辑

### 6.1 进入页面

1. 从 URL 读取 `chapter`（缺省为 1），与 store 的 `currentChapter` 同步。
2. 若 store 中 `opponent` 为空，视为异常（如直接访问 `/game`），可重定向到选择对象页或首页。
3. 若 `currentNodeId` 为空，则根据 `chapter` 取该章**第一个节点**（如 `story.json` 中 `chapter === 1` 且顺序第一的节点）作为当前节点并写入 store。
4. 加载 `story.json`，根据 `currentNodeId` 取当前节点，渲染剧情与选项。

### 6.2 文本插值

- 将节点 `text` 中的占位符替换为实际值，例如：
  - `{opponent.name}` → 当前相亲对象姓名
  - `{opponent.mbti}` → 对方 MBTI
  - 若有 `{player.xxx}` 可按需扩展（如年龄、职业描述）。
- 未定义占位符可替换为空或保留原样，避免报错。

### 6.3 选项条件过滤

- 渲染前对 `options` 做过滤：若选项带 `condition`（如 `affection <= 60`），则根据当前 store 中的 `affection` 判断是否展示该选项。
- 仅展示通过条件的选项；若全部不通过，可展示一条默认选项（如「继续」）或提示「暂无可选」，由实现与产品约定。

### 6.4 选项点击与效果执行

1. **效果执行**：按选项的 `effects` 依次执行：
   - `type: "affection", operator: "add"`：若存在 `value` 则直接 `addAffection(value)`；若存在 `valueByMbti` 则按 `opponent.mbti` 取增量后 `addAffection`。
2. **跳转**：根据 `nextNode` 在 `story.json` 中查找节点：
   - **找到**：更新 store 的 `currentNodeId` 与 `currentChapter`，并可选更新 URL 的 `chapter`，重新渲染当前节点。
   - **未找到（单节点占位）**：不修改 `currentNodeId`，可弹出 Toast 或页面内提示「本章未完待续」，并保持当前节点与好感度（好感度已在点击时加上）。

### 6.5 单节点开发说明（当前剧本仅 c1_node1）

- 当前 `story.json` 仅包含一个节点 `c1_node1`，其选项的 `nextNode` 指向 `c1_node2`、`c1_node3`，剧本中暂无对应节点。
- **约定**：点击任选项后照常执行好感度效果，然后检测下一节点是否存在；若不存在，走「单节点占位」逻辑（提示 + 停留当前节点），不报错、不白屏。
- 后续补充 `c1_node2`、`c1_node3` 等节点后，仅需保证数据加载与跳转逻辑一致即可，无需改交互设计。

## 7. 与状态管理及本地存储的衔接

- **Zustand / Redux**：`currentChapter`、`currentNodeId`、`affection`、`opponent`、`flags` 与现有 `useGameStore` 一致；选项效果通过 `addAffection`、`setFlag`、`setCurrentNode` 更新。
- **本地存储**：按需求文档 3.1，游戏过程中可自动保存当前章节与节点，刷新后从 store 或 localStorage 恢复进度；具体 key 与读写时机可与现有「角色信息」存储方案统一。

## 9. 与现有设计的一致性

- **色彩与风格**：MBTI 代表色、毛玻璃、圆角与《视觉设计》文档一致（ENFJ 暖橙、INTJ 冷紫、INFJ 森林绿、ENTJ 酒红）。
- **字体与组件**：标题、按钮、卡片与首页、创建角色模块统一；选项按钮 hover/点击反馈（缩放、高亮）保持一致。
- **资源复用**：背景、卡片、人物图与首页、创建角色共用同一套 public 资源。
