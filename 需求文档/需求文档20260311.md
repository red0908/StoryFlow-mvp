# 产品需求设计文档：MBTI 相亲互动游戏（MVP）

## 1. 产品概述
### 1.1 产品背景
一款轻量级、情景式的相亲互动游戏，引入 MBTI 人格理论，让玩家在模拟相亲过程中体验性格匹配的乐趣。MVP 版本将上线四种人格（ENFJ、INTJ、INFJ、ENTJ），通过四个章节的剧情分支，让玩家感受不同性格角色之间的互动张力。

### 1.2 核心价值
- **娱乐 + 心理**：将 MBTI 与剧情选择深度融合，每次选择都会影响好感度与剧情走向。
- **轻量快速**：无需后端，纯前端实现，一周内可上线验证核心玩法。
- **可扩展**：剧本与角色数据均采用 JSON 配置，后续可轻松增加人格类型、章节与结局。

## 2. 用户流程与核心玩法
### 2.1 整体流程
```
启动游戏 → 创建个人角色 → 浏览异性候选人（5选1） → 进入四章剧情（开头/发展/高潮/结束） → 达成结局
```
- 每章包含 **4~5 个剧情节点（场景）**，每个节点提供 2~3 个选项，选项导致好感变化或分支跳转。
- 最终根据累积好感值或关键选择触发不同结局（如“灵魂伴侣”“普通朋友”“不欢而散”）。

### 2.2 角色创建
| 字段 | 内容 | 说明 |
|------|------|------|
| 性别 | 男 / 女 | 用于匹配异性候选人 |
| 年龄 | 25~35 岁（滑动条/下拉） | 仅记录，剧情中可引用 |
| 职业 | 5 种预设（如：医生/教师/工程师/设计师/律师） | 影响初始介绍文本 |
| MBTI | 从 ENFJ/INTJ/INFJ/ENTJ 中单选 | 附带一句人格描述（如“ENFJ：热情的教育者”） |
| 一句话介绍 | 自由填写或系统生成 | 用于展示在候选人界面 |

### 2.3 选择相亲对象
- 系统随机生成 **5 位异性候选人**（玩家性别为男则生成女性，反之男性）。
- 每位候选人包含：
  - 姓名（随机池）
  - 年龄（25~35 随机）
  - 职业（从 5 种职业中随机）
  - MBTI（从 ENFJ/INTJ/INFJ/ENTJ 中随机）
  - 一句话介绍（根据 MBTI 及职业组合生成，例如“INTJ 建筑师，喜欢理性探讨”）
- 玩家点击其中一位，即锁定该角色为本次相亲对象，进入剧情。

### 2.4 剧情互动
- 界面分为：
  - **顶部状态栏**：显示当前章节（如“第一章·开头”）、双方 MBTI 图标、当前好感度（数值或进度条）。
  - **中部剧情框**：显示当前节点的描述文本（可动态插入双方姓名、年龄、MBTI 等信息）。
  - **底部选项区**：显示 2~3 个可点击的选项按钮，点击后触发效果并跳转至下一节点。
- 每个选项可导致：
  - 好感度增减（可针对不同对方 MBTI 设置不同变化值）
  - 直接跳转到指定节点
  - （可选）解锁隐藏变量，影响后续选项出现

### 2.5 结局系统
- 第四章最后一个节点结束后，根据好感度范围及关键选择展示对应结局画面与评语。
- 预设 3~4 种结局，例如：
  - **完美匹配**（好感 ≥ 80）：双方性格契合，相亲成功。
  - **略有遗憾**（好感 50~79）：有好感但未进一步发展。
  - **分道扬镳**（好感 < 50）：性格不合，相亲失败。
  - 隐藏结局：特定选项组合触发（如“成为事业伙伴”）。

## 3. 功能需求详情
### 3.1 初始化与数据持久化
- 使用 `localStorage` 保存玩家最近一次的角色信息，方便快速开始新游戏。
- 游戏过程中自动保存当前章节与节点，刷新后回到最近进度。

### 3.2 随机候选人生成
- 维护姓名库（男/女各 10 个常用名），职业库，MBTI 权重平均。
- 每次进入选择界面重新随机生成，确保新鲜感。

### 3.3 剧本驱动引擎
- 所有剧情节点、选项、效果均以 JSON 格式定义。
- 引擎核心功能：
  - 根据当前节点 ID 加载文本与选项
  - 执行选项绑定的效果（好感增减、变量赋值）
  - 条件判断（如某些选项仅当好感 > 50 或玩家 MBTI 为某类型时才出现）
  - 节点跳转，支持跨章节跳转

### 3.4 章节与节点管理
- 每个节点属于唯一章节（chapter 1~4）。
- 节点数量：每章固定 4~5 个节点，总节点数 ≤ 20。
- 选项可指向同一章节的后续节点，也可指向下一章节的第一个节点（使用 `nextChapter` 标识）。

### 3.5 好感度计算
- 初始好感 = 基础值 50。
- 选项效果可定义为固定增减，或根据对方 MBTI 动态增减（例如：“理性分析”选项对 INTJ 好感 +5，对 ENFJ 好感 +1）。
- 界面实时刷新好感变化。

## 4. 系统架构
### 4.1 总体架构图
```
[前端 SPA] (React + Redux Toolkit)
  ├── 视图层：创建角色 / 选择对象 / 剧情互动 / 结局展示
  ├── 状态管理：playerSlice, gameSlice (好感/当前节点/历史)
  ├── 剧本引擎：剧本 JSON 加载器、条件解释器、效果执行器
  └── 工具层：随机生成器、本地存储封装
```
- **无后端**，所有数据（剧本、角色库）打包在静态资源中。
- 部署于 Vercel / GitHub Pages，纯静态托管。

### 4.2 模块划分
| 模块 | 职责 |
|------|------|
| 角色创建模块 | 收集用户输入，校验并存储至 Redux store |
| 候选人模块 | 生成 5 个异性对象，提供选择交互 |
| 剧情主控模块 | 加载剧本，渲染当前节点，处理选项点击 |
| 剧本数据 | 剧本 JSON 文件，包含节点、选项、条件、效果 |
| 本地存储模块 | 读写游戏进度与角色信息 |

## 5. 技术方案
### 5.1 技术栈
- **构建工具**：Vite
- **框架**：React 18 + Redux Toolkit
- **样式**：Tailwind CSS（快速搭建 UI，减少手写 CSS）
- **路由**：不使用 react-router，采用条件渲染切换（创建/选择/游戏/结局）
- **类型检查**：TypeScript（可选，可提高代码健壮性）
- **部署**：Vercel / Netlify

### 5.2 关键数据结构
#### 玩家信息 (player)
```typescript
interface Player {
  gender: 'male' | 'female';
  age: number;
  job: string;        // 职业ID
  mbti: 'ENFJ' | 'INTJ' | 'INFJ' | 'ENTJ';
  description: string; // 一句话介绍
}
```

#### 候选人信息 (Candidate)
```typescript
interface Candidate {
  id: string;
  name: string;
  age: number;
  job: string;
  mbti: 'ENFJ' | 'INTJ' | 'INFJ' | 'ENTJ';
  tagline: string;     // 一句话介绍
}
```

#### 游戏状态 (game)
```typescript
interface GameState {
  opponent: Candidate;         // 当前选择的相亲对象
  currentChapter: number;      // 1-4
  currentNodeId: string;       // 剧本中的节点ID
  affection: number;           // 好感度 0-100
  flags: Record<string, any>;  // 自定义变量（用于复杂分支）
}
```

#### 剧本节点 (StoryNode)
```json
{
  "id": "c1_node1",
  "chapter": 1,
  "text": "你坐在咖啡厅，等待 {opponent.name} 的到来。她/他穿着一件简单的衬衫，微笑着向你打招呼。",
  "options": [
    {
      "text": "直接开门见山，聊聊工作。",
      "nextNode": "c1_node2",
      "effects": [
        { "type": "affection", "operator": "add", "value": 2 }
      ]
    },
    {
      "text": "先夸赞对方的穿着，拉近距离。",
      "nextNode": "c1_node3",
      "effects": [
        { 
          "type": "affection", 
          "operator": "add", 
          "valueByMbti": { "INTJ": 1, "ENFJ": 3, "INFJ": 2, "ENTJ": 1 }
        }
      ],
      "condition": { "type": "affection", "operator": "<=", "value": 60 }  // 例：好感≤60时出现
    }
  ]
}
```

### 5.3 条件与效果解释器
- **条件支持**：`affection`（好感比较）、`playerMbti`、`opponentMbti`、`flag`（自定义变量）。
- **效果支持**：
  - `affection` 增减（固定值或按对方 MBTI 映射）
  - `setFlag` 设置自定义变量
  - `unlockOption`（预留，后续扩展）

## 6. 剧本架构设计
### 6.1 剧本总量规划（MVP）
- **总节点数**：20 个（每章 5 个节点，共 4 章）。
- **分支复杂度**：每个节点 2~3 个选项，关键节点可产生 2 种不同后续路径，最终在章节末汇合或引向不同结局。
- **好感影响**：每个选项至少包含一条好感效果，体现 MBTI 差异（例如通过 `valueByMbti` 映射）。

### 6.2 剧本示例（第一章·开头）
| 节点 ID | 场景描述 | 选项 | 效果 |
|--------|----------|------|------|
| c1_node1 | 初次见面，互相介绍。 | A. 幽默开场<br>B. 正式自我介绍<br>C. 询问对方兴趣 | A: 对ENFJ+3，INTJ+0<br>B: 对INTJ+3，ENFJ+1<br>C: 通用+2 |
| c1_node2 | 聊到了工作压力…… | …… | …… |

*实际剧本将由产品经理或策划按照此结构填充，开发期间可使用占位文本测试流程。*

### 6.3 MBTI 差异化实现
- 不在剧本中为每种 MBTI 编写独立文本，而是通过 **效果差异化 + 文本插值** 体现。
- 文本中可插入 `{opponent.mbti}` 或根据 MBTI 显示不同性格描述（例如使用 `v-if` 在模板中判断，但建议保持简单，初期仅通过好感变化让玩家感知差异）。

| **Day 4** | 填充完整剧本内容（文本、选项、效果） | 所有节点具备可玩剧情，MBTI 差异化效果初步植入 |

## 8. 风险与应对
| 风险 | 应对 |
|------|------|
| 剧本编写量超出预期 | 先保证核心主线完整，支线用简单分支代替；后续迭代扩充 |
| MBTI 差异化导致选项爆炸 | 初期仅通过好感数值差异化实现，文本差异化靠模板插值 |
| 节点跳转逻辑复杂 | 设计平面化的节点 ID 命名规则（如 `c1_n1`），并绘制简单流程图 |
| 移动端适配问题 | 使用 Tailwind 的响应式类，确保在手机屏幕上布局正常 |
